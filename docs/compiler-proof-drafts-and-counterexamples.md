This note preserves useful documentation from deleted HOL theory scripts that were not part of the active compiler proof dependency graph.

Deleted source theory scripts:

- `venom/compiler/compilerCorrectnessDraftScript.sml`
- `venom/codegen/proofs/counterexamplesScript.sml`

## Compiler correctness architecture draft

The intended end-to-end compiler correctness argument has three legs:

```text
Vyper source
  -- lowering -->
Venom IR
  -- optimization pipeline -->
Optimized Venom IR
  -- codegen -->
EVM bytecode
```

The draft theory described these proof obligations:

1. **Lowering correctness**: Vyper source semantics (`call_external`) corresponds to Venom execution (`run_context`) on the context produced by lowering.
2. **Pipeline correctness**: optimizing Venom transformations preserve Venom semantics, via context/pass correctness composition.
3. **Codegen correctness**: Venom execution corresponds to EVM execution of generated bytecode.
4. **EVM call bridge**: connect the EVM execution relation used by codegen to the external-call correspondence used by the Vyper/EVM top-level theorem.

Current active theories corresponding to this structure include:

- `lowering/vyperLoweringCorrectScript.sml`
- `lowering/e2eCorrectnessScript.sml`
- `venom/compiler/venomPipelineCorrectScript.sml`
- `venom/codegen/venomToAsmPropsScript.sml`
- `venom/codegen/asmToBytecodePropsScript.sml`
- `venom/codegen/codegenCorrectnessScript.sml`

### Draft lowering theorem shape

The deleted draft stated a theorem of the following shape:

```sml
run_lowering ... = (ctx, ...) /\
valid_function_call ...
==>
external_call_result_rel ...
  (call_external am tx)
  (run_context fuel ctx vs)
```

The noted missing proof work was:

- ABI encoding/decoding: calldata encoding matches Venom calldata/memory operations.
- Expression lowering: `compile_expr` produces correct Venom instruction sequences.
- Statement lowering: `compile_stmt` preserves source/target state correspondence.
- Function dispatch: selector matching and Venom function lookup agree with source calls.
- Storage operations: Vyper storage load/store corresponds to Venom `SLOAD`/`SSTORE`.
- Memory operations: Vyper memory/heap model corresponds to Venom `MLOAD`/`MSTORE`.
- Log emission: Vyper logs correspond to Venom `LOG` instructions with correct topics/data.
- Control flow: Vyper control flow corresponds to Venom `JMP`/`JNZ`/`STOP`/`REVERT`.

### Draft codegen context obligations

The deleted draft isolated obligations needed before applying codegen correctness:

- `codegen_ready ctx`
- `ctx_wf ctx`
- entry function does not return via internal-return mechanism
- all Venom steps preserve the spill/memory-safety region needed by codegen

These should remain explicit obligations in active codegen/e2e proof statements rather than hidden in compiler composition.

### Draft pass obligations

The deleted draft listed sample pass-correctness obligations for pass-specific theories, including:

- algebraic optimization
- branch optimization
- single-use expansion
- affine folding
- overflow elimination

The active generic composition infrastructure is in `venom/compiler/venomPipelineCorrectScript.sml`; individual pass correctness theorems should feed into that framework.

## Codegen counterexamples and proof lessons

The deleted `counterexamplesScript.sml` documented concrete issues with earlier/frozen theorem statements.

### `gen_inst_simulation` false for `PARAM`

A `PARAM` instruction updates `vs_vars` from `vs_params`, but generates no assembly instructions. If a variable is already present on the modeled asm stack with an old value, then after the Venom `PARAM` step, `plan_stack_rel` can break because the asm stack still contains the old value.

Sketch:

- instruction: `PARAM [Lit 0w] -> ["x"]`
- initial Venom vars: `x = 7w`
- params: `[42w]`
- plan stack: `[Var "x"]`
- asm stack: `[7w]`
- generated plan for `PARAM`: `[]`
- Venom step updates `x` to `42w`
- asm stack remains `[7w]`
- `plan_stack_rel` fails after the step

Lesson: per-instruction simulation must either exclude `PARAM` or include stronger block/function-level invariants explaining how parameters are prepared. The old note suggested `inst_opcode <> PARAM`; block-level code already filters/handles params separately in some paths.

### Operand order bug in `reorder_one_def`

The note recorded a historical operand-order issue: a formula using

```text
final_dist = num_ops - 1 - target_idx
```

matches a Python push-order convention, but HOL operands are in EVM semantic order. For opcodes like `SUB [a,b]`, this can put `b` at TOS and `a` below it, producing `b - a` instead of `a - b`.

The recorded fix was to use:

```text
final_dist = target_idx
```

This should be checked against the current `stackPlanOpsScript.sml` before treating the issue as live.

### `asm_bytecode_sim` missing preconditions

The note recorded that an earlier `asm_bytecode_sim` statement was false without extra preconditions:

1. `no_asm_calls prog` — the asm interpreter handles calls in a single-context style, while EVM execution may push/pop call frames.
2. `LENGTH es.contexts = 1` — with multiple EVM contexts, `STOP` in an inner context returns to the outer context, whereas asm `STOP` halts unconditionally.

Lesson: asm/EVM simulation theorem statements must state enough restrictions about call handling and context depth, or explicitly model nested-frame behavior.

## Lowering memory-safety counterexamples

The deleted `loweringAllocaSafeAccessCounterexampleScript.sml` documented false earlier theorem statements about lowering memory safety.

### Counterexample 1: alloca invariant alone is insufficient

The counterexample constructed a lowered function containing:

- an `ALLOCA [Lit 32w]` instruction producing `%21`
- an `MSTORE [Var "%21"; Lit 0w]`

Then it supplied an adversarial runtime state where:

- `vs_allocas` maps the relevant allocation id to a region of only one byte, e.g. `(offset=0, size=1)`
- `%21 = 0w`
- memory length is large enough globally

This can satisfy generic allocation well-formedness/non-overlap facts while failing the specific safety condition needed for a 32-byte `MSTORE`.

Lesson: a theorem connecting lowered ALLOCA sizes to runtime `vs_allocas` must require a state/function matching hypothesis, not just an arbitrary `alloca_inv` on the state.

### Counterexample 2: size agreement alone is insufficient

A second counterexample kept ALLOCA sizes consistent but assigned the alloca output variable an out-of-region address.

Lesson: matching allocation sizes is not enough; runtime pointer values stored in variables must also be constrained to point to the corresponding allocation region.

### Relevant active theories

The current memory-safety proof work should be checked in:

- `lowering/defs/loweringMemSafetyDefsScript.sml`
- `lowering/loweringMemSafetyPropsScript.sml`
- `lowering/proofs/loweringMemSafetyProofsScript.sml`

The counterexamples motivate hypotheses such as:

- compile environment ALLOCA sizes match the function,
- runtime ALLOCA table matches the function,
- ALLOCA output variables contain the runtime base addresses,
- pointer-derived variables remain within their allocation bounds.
